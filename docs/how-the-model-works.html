How the model works, from first principles

You already know attention. This is the other half of the deep learning world: how a convolutional network turns a wrecked raster into a clean answer key for the Rust engine, and where the one open design decision lives.

10 min read Assumes: attention basics vecml, Phase 0
The whole machine in one sentence The model is a single learned function: it takes a 256 by 256 grid of damaged pixels and outputs, for every one of those pixels, which flat region of the original artwork it belongs to. Everything below is detail on how that function is built and trained.

We will follow one image through the entire machine: the specimen above, the file you flagged an hour ago. Two ellipses of the same translucent blue, whose overlap creates a third, darker colour. By the end you will know exactly what happens to its numbers at every stage, and why the machine is shaped the way it is.

STAGE 1An image is a very different tensor from a sentence

An LLM eats a sequence: a few thousand tokens, each already a rich embedding, and attention lets every token consult every other. Our input is a picture: 256 × 256 × 3, which is 65,536 pixels of three small numbers each (red, green, blue, scaled to roughly 0 to 1). No embeddings, no vocabulary. The raw numbers are the input.

You could flatten those pixels into 65,536 tokens and run a transformer. Attention compares every position with every other, so that is about 4.3 billion pairwise comparisons per layer, spent mostly asking whether a pixel in the top-left corner is relevant to one in the bottom-right. For pictures that is the wrong question to ask first, because images have a property text does not have nearly as strongly: locality. The evidence for what a pixel is sits overwhelmingly in the pixels touching it. A blurry edge is diagnosed from its neighbourhood, not from across the canvas.

STAGE 2Convolution: the same small question, asked everywhere

Attention learns where to look: routing is dynamic, computed per input. Convolution makes the opposite bet: it decides what to look for is the same everywhere, and hard-wires the looking. A convolutional filter is a tiny grid of learned weights, typically 3 by 3, and it is slid across every position of the image. At each position you multiply the window of pixels by the filter, sum the result, and write one number to an output grid. That output number means "how strongly does this exact spot match my pattern".

Three consequences fall out of this design, and they are why convolution owns vision:

  • Weight sharing. One filter is 9 numbers no matter how big the image is. A layer of 32 filters over our whole specimen costs about 300 weights. The equivalent fully-connected layer would need billions.
  • Translation equivariance. An edge detector fires on edges wherever they appear. The network does not need to relearn "curve" separately for every location on the canvas, which is exactly right for logos that can sit anywhere.
  • Locality matches the damage. JPEG blocks, blur, and noise are all local corruptions. The evidence needed to undo them is local too.

One filter produces one output grid, called a feature map. A layer runs many filters in parallel, so its output is a stack: 256 × 256 × 32 after the first layer. The specimen is no longer red, green, blue at each pixel; it is now 32 numbers per pixel meaning "edge here", "blue-ish here", "smooth here". Stack layers and the vocabulary compounds: layer one sees edges, layer two sees corners and arcs built from those edges, layer three sees whole motifs. Same idea as deep layers of a transformer building from syntax to meaning, but grown outward from neighbourhoods instead of routed by attention.

STAGE 3The U: trade where for what, then buy where back

Here is the tension that dictates the architecture. To decide what the specimen is, some part of the network must see all of it at once: "two overlapping ellipse-like blobs on white". But a 3 by 3 filter sees a 3 by 3 neighbourhood. The fix is to shrink the grid as you go deeper: halve the resolution every couple of layers while doubling the number of feature maps. Each cell of a smaller grid summarises a larger patch of the original, so by the bottom of the network one cell's view covers the whole picture.

That solves understanding and breaks output. Our answer must be pixel-precise: the label map is judged at full resolution, and a boundary off by three pixels is a wrong answer key. So after descending, the network ascends: upsample the grid back, halving the feature count as resolution returns. And because the descent threw away fine position information on purpose, the ascent gets it back through skip connections: at each resolution, the feature maps from the way down are handed directly across to the way up, carrying exact edge positions the bottleneck never kept. Drawn honestly, the architecture is a U.

Where your attention knowledge plugs in Notice the base of the U: a 32 by 32 grid is only 1,024 positions. That is transformer territory. Many modern segmentation networks run a few attention layers exactly there, where global reasoning is cheapest, and keep convolutions everywhere else. If we ever add attention, that is the slot. The pure-convolution U-Net comes first because it is small, fast on the M5, and the classic baseline.

STAGE 4The output problem: a network cannot say "region 3"

Now the honest open question, and the reason the first training run is designed to force a decision. Our answer keys say things like "pixel (40, 61) belongs to region 3 of this file". That works per file, but region numbers are arbitrary per image. A network's output layer is fixed at design time; there is no neuron that can mean "region 3" across a million different images, each with its own regions. So the model must express region membership some other way, and there are three standard candidates.

HeadThe model outputs, per pixelTrade-off
Colour regression The clean RGB colour it believes belongs there Simplest, but averaging is the failure mode: unsure between blue and white at a boundary, the loss rewards guessing pale blue, a colour that exists nowhere in the art. Muddy edges are precisely what we are trying to eliminate.
Embedding + clustering A short vector; training pulls vectors of same-region pixels together and pushes different-region pixels apart Handles any number of regions per image. At inference you cluster the vectors to recover regions. The mainstream choice for this problem shape, at the cost of a clustering step.
Boundary + fill The probability that a region border passes through this pixel Regions come free afterwards by flood-filling between borders, which is very friendly to a tracing engine. But a single one-pixel gap in a predicted border merges two regions, so errors are catastrophic rather than graceful.

Our specimen is a sharp test for all three. The overlap lens is a genuinely different colour than the lobes, so colour regression must not average it away; the embedding head must push lens pixels apart from lobe pixels despite both being blue; the boundary head must draw the lens outline even though no single source shape has that outline. This decision, not GPU shopping, is what the overfit run is for.

STAGE 5Training: the same loop you already know, with a different exam

Mechanically, training this model is the same three-step loop as training an LLM. Only the tensors differ.

  1. Forward: feed the wrecked specimen through the U, get a predicted map.
  2. Loss: compare prediction against the answer key from the wrecking pipeline, pixel by pixel, and reduce all 65,536 verdicts to one number for how wrong the model was. An LLM's loss asks "did you predict the next token"; ours asks "did you label each pixel".
  3. Backward: backpropagation runs the chain rule in reverse to assign each of the network's weights its share of the blame, and the optimiser (Adam) nudges every weight slightly against its blame. Identical machinery, identical code path in PyTorch.

The difference that matters is the exam supply. An LLM trains on found text. We manufacture our exams: the wrecking pipeline takes a clean SVG, renders it, damages it with a known recipe, and keeps the undamaged answer key. Every pair teaches the model one instance of the general lesson "this kind of mush, on this kind of art, hides these exact shapes". A million pairs later, the lesson is a prior: the model has seen JPEG ringing around two-colour boundaries so many times that it treats it as damage to see through rather than texture to preserve. That prior is the entire advantage over classical methods, which meet every image with no memory of any other image.

The classical engine reasons about the pixels in front of it. The model reasons with a million images it has already seen.

Why a learned front-end can beat any hand-written heuristic on damaged input
Why the first run deliberately overfits The first training run uses about 100 images and trains until the model reproduces their answer keys almost perfectly. Memorisation, not learning, and that is the point: a network that cannot even memorise 100 images has a plumbing bug (data loading, loss wiring, label format), while one that can proves the whole loop end to end. It is the "hello world" of training runs, it is free, and it runs on the M5 in minutes.

STAGE 6The relay: model eyes, engine hands

The model never draws a curve. Its output for the specimen is a cleaned, labelled map: these pixels are lobe one, these lobe two, these the darker lens, everything else background. That map goes to the Rust engine, which does what it already does well: trace exact shared boundaries through the planar map, fit curves, and emit SVG. The two never blur together. The model is not inside the engine and the engine is not inside the model; the model hands over a finished decision about what shapes exist, the engine turns that decision into geometry.

This is also why model loss is not our success metric. Loss measures pixels; the product is geometry. A model can nudge loss down by getting easy background pixels slightly more confident while still botching the one boundary that matters. So every candidate model gets judged the same way your classical attempts always have: run the full relay, score the final SVG with veval (colour error, reconstruction quality, holes), and compare it in the gallery against flat_quant on wrecked inputs it has never seen. The model earns its place in the pipeline the day the relay beats the engine alone.

And that closes the loop on why your audit work this week mattered so much. Every mislabeled answer key, the deleted Python logo, the missing overlap lens, would have been taught to the model as truth, a million times over. The network is only ever as honest as its exams.
