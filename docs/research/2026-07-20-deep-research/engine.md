# Theme 3 — Engine Gap: why our Rust planar-map tracer loses to vtracer

Research date: 2026-07-20
Scope: the 0.16-deltaE ceiling gap on *perfect* input (our Rust engine 1.44 vs vtracer 1.28) and the wider gap on identical model-cleaned input (Rust 1.79 vs vtracer 1.52). Sources are vtracer/visioncortex source code, the Selinger 2003 Potrace paper, the maptrace practitioner writeup, and line-simplification/subpixel-edge literature. Each decision-critical claim was adversarially re-checked against primary source; corrections are folded in below.

---

## 1. Executive summary

The gap is a **curve/polygon-fitting fidelity** problem, not a topology problem and not a subpixel-edge-reading problem. Two independent search angles converged on the same conclusion: both vtracer and Potrace destroy anti-aliasing/coverage before they fit any curve, so neither reads subpixel edges — meaning our loss on *perfect* input cannot be blamed on edge localisation. What separates the tracers is how well they turn a pixel-staircase boundary into a clean polygon and then into Beziers.

Three levers stand out, in order of expected payoff:

1. **Adopt Potrace's boundary-approximation machinery.** Potrace's defining quality step is a *globally optimal polygon* (fewest segments, then lowest penalty, found by shortest-cycle graph search) followed by *sub-pixel vertex adjustment* (least-squares fit-line intersection, constrained to a 1/2-pixel box). vtracer — which already beats us — uses only a *greedy* penalty simplifier and no global optimum. If our engine uses plain Douglas-Peucker or snaps vertices to the integer grid, we are below even vtracer's bar. The penalty metric, vertex adjustment, corner/curve analysis and curve-merge stages are all colour-agnostic and can wrap a planar-map boundary graph; only Potrace's path decomposition and four-direction straightness test are bilevel-specific and must be replaced with a polyline-native scorer.

2. **Guarantee shared-edge consistency by construction.** vtracer's default *stacked* mode sidesteps seams entirely by drawing overlapping opaque layers; its *cutout* mode (architecturally what a true planar map is) admits it "does not share boundaries perfectly, but has seams." A planar-map engine must therefore trace each shared arc *once* on a half-edge/PSLG graph and have both adjacent faces reference the identical simplified polyline. Independently simplifying each face and reconciling later is the documented failure mode (measured 4.5–68.9% inter-region overlap vs 0% for a globally-consistent partition).

3. **Use mean-of-residue fill and protect corners before smoothing.** vtracer fills each region with the arithmetic mean of its residual pixels (exact colour recovery on flat regions, deltaE ~0 by construction) and hard-gates corners (turn angle >= 60 deg) so smoothing never rounds them. If our engine palette-snaps fills or smooths uniformly, both cost deltaE precisely at the boundaries where our label errors already concentrate.

A caveat on measurement: mean-deltaE over the icon-heavy 24-image bench is area-dominated and nearly blind to half-pixel edge error, so fitting-quality gains will read clearly here while any future subpixel work will barely move this bench and only pay off on typography/fineline content. Longer term, vectorizer.ai's edge over both tracers is a richer primitive set (arcs, ellipses, parameterised shapes, symmetry) that a cubic-Bezier-only engine cannot reach — out of scope for closing the current 0.16 gap but relevant to the ceiling beyond it.

## 2. Findings by sub-topic

### 2.1 The gap is fitting quality, not edge reading

Neither reference tracer positions edges from anti-aliased coverage. Potrace operates on a pure black/white bitmap and places path vertices at integer pixel *corners*; its later "vertex adjustment" is polygon optimisation, not coverage reading ([Potrace paper](https://potrace.sourceforge.net/potrace.pdf)). vtracer colour-quantises first (discarding anti-aliasing), then walks integer-step cluster outlines and removes staircases by signed area ([vtracer docs](https://www.visioncortex.org/vtracer-docs/)). Both discard subpixel coverage *before* any boundary refinement, so the measured perfect-input gap (1.28 vs 1.44) tracks the fitting stage, and the same ~0.16–0.27 deltaE gap persists on both perfect and cleaned input — consistent with a fitting deficit rather than an input-quality one.

Subpixel-from-coverage is nonetheless a real, quantified, *untapped* differentiator (pixel-grid detection bounded at ~0.5px error; interpolation methods reach ~0.02–0.05px median — [arXiv 2502.16502](https://arxiv.org/pdf/2502.16502)), and recovering subpixel geometry from anti-aliased clip-art is validated prior work ([Yang et al., CGF 2023](https://onlinelibrary.wiley.com/doi/10.1111/cgf.14744)). But it will not explain or close the *current* loss, and it needs a soft/coverage signal that our 16-class argmax label head currently throws away. Treat it as a later ceiling-raiser, not a fix for the 0.16 gap.

### 2.2 Potrace's boundary machinery (the main lever)

Potrace's quality comes from a **globally optimal polygon**: it finds the polygon with the fewest segments, then the lowest total penalty, by solving for the optimal cycle in a directed graph — an O(n·m), explicitly *non-local* operation where "each part of the optimal polygon depends potentially on all the other parts" ([Potrace paper](https://potrace.sourceforge.net/potrace.pdf)). vtracer, which beats us, uses only a *greedy* penalty-based simplification (extend a subpath until max penalty exceeds tolerance, then cut), O(n) end-to-end ([vtracer docs](https://www.visioncortex.org/vtracer-docs/)). That gap is the actionable lever: if we are greedy or DP-based, we are at or below vtracer's tier and can leapfrog it with the global optimum.

The penalty metric is directly implementable and O(1) per candidate segment via prefix sums: `P(i,j) = |v_j - v_i| * stddev of perpendicular distances of enclosed points from the chord`, with a closed form from the covariance sums ([Potrace paper](https://potrace.sourceforge.net/potrace.pdf)). A clipping rule matters for quality: a candidate segment i→j is valid only if the subpath extended by one point on *both* ends stays straight — Selinger notes that without this there is "strange behavior around the corners."

Potrace also does **sub-pixel vertex adjustment** before curve fitting: for each polygon edge it least-squares-fits a line to the underlying path points (best-fit slope = eigenvector of the larger eigenvalue of the covariance matrix), then places each vertex at the intersection of its two adjacent fit-lines, constrained to a unit square around the original vertex ([Potrace paper](https://potrace.sourceforge.net/potrace.pdf)). If our engine leaves boundaries on integer pixel corners, this is a concrete missing refinement.

Corner/curve decision is a single scalar `alpha = 4·gamma/3`, clamped to [0.55, 1]: `alpha <= alphamax` (default 1) gives a smooth Bezier, above gives a hard corner; the 0.55 lower bound is the optimal quarter-circle Bezier fit (deviation <0.014%). An optional curve-optimisation stage merges adjacent Beziers of matching convexity (total turn <179 deg), cutting segment count ~40% with no visible change (tolerance `opttolerance` default 0.2).

**Reusability boundary (verified):** only path decomposition (recursive pixel inversion) and the four-direction straightness test are bilevel-specific and cannot be lifted to arbitrary planar-map boundary polylines. Vertex adjustment, penalty scoring, corner/curve analysis and curve optimisation take "a polygon"/"a closed path" as input and are colour-agnostic — they can wrap a planar-map boundary graph if decomposition + straightness are replaced with a polyline-native subpath scorer.

**Resolution caveat:** Potrace targets high-resolution input and its own paper warns corner analysis "works better at higher resolutions"; "no tracing algorithm will work well on very small scales." We trace at 256px — the regime where corner detection is weakest. Tracing the cleaned raster at higher resolution (or before any downscale) may matter as much as the tracer itself.

### 2.3 Shared-boundary consistency (structural, planar-map-specific)

This is the hard regime vtracer *avoids*. Its default **stacked** hierarchical mode draws each colour layer as a full opaque shape back-to-front, so adjacent regions overlap underneath and there is never a seam; the overlap hides the anti-aliased boundary band under the top layer ([vtracer converter.rs](https://github.com/visioncortex/vtracer/blob/master/cmdapp/src/converter.rs)). Its **cutout** mode — architecturally what a true planar map is — explicitly "does not share boundaries perfectly, but has seams" ([vtracer README](https://raw.githubusercontent.com/visioncortex/vtracer/0.6.4/README.md)). So our engine competes in the regime vtracer sidesteps by construction.

> Verification note: the characterisation of stacked order as "largest-area-first" is the researcher's description of the *effect*; the primary source shows layering follows hierarchical cluster build order (parents/backgrounds first), and an explicit area-sort could not be confirmed in the code. The overlap mechanism and the Stacked default are confirmed verbatim. The "single biggest structural reason" phrasing is an interpretive superlative, directionally supported but not quantified by the source.

The literature quantifies the cost of getting the true-partition case wrong: independently traced/simplified region boundaries produce 4.5–68.9% inter-region overlap versus 0.0% for a globally-consistent planar-straight-line-graph reconstruction ([ACPV-Net, arXiv 2603.16616](https://arxiv.org/html/2603.16616v1)). The fix is structural and named: build one planar graph (half-edge/PSLG), trace and simplify each shared arc exactly once, and have both adjacent faces reference the identical simplified polyline — the TopoJSON/GEOS topology-preserving approach, versus the standard warning that per-polygon simplification "will easily get gaps between polygons" ([Fleischmann, line-simplification](https://martinfleischmann.net/line-simplification-algorithms/)). The post-hoc alternative (Delaunay-detect the inconsistent strip, build a medial-axis unified boundary) exists but is more complex than getting it right by construction ([WHU paper](https://ch.whu.edu.cn/en/article/id/5124)).

### 2.4 The simplifier itself — what NOT to use

vtracer's simplifier is neither Douglas-Peucker nor Visvalingam-Whyatt: it is a greedy penalty-based line-replacement (penalty proportional to triangle height and base) plus signed-area staircase removal, corner detection *before* smoothing, and 4-point subdivision with corner preservation and inflection/angle-displacement spline splicing ([vtracer docs](https://www.visioncortex.org/vtracer-docs/)). If our engine uses plain Douglas-Peucker it inherits DP's known defects: spiky artefacts at certain configurations and possible self-intersecting (non-simple) output; VW yields smoother geometry but is ~30x slower (74.1us vs 2.17ms/loop) and both can self-intersect without a topology-preserving variant ([Fleischmann](https://martinfleischmann.net/line-simplification-algorithms/); [RDP Wikipedia](https://en.wikipedia.org/wiki/Ramer%E2%80%93Douglas%E2%80%93Peucker_algorithm)). For icon-dominated, corner-heavy content, DP's corner-spike issue is the more damaging failure. A pre-merge step — vtracer combines consecutive same-direction pixel steps before simplification — feeds the simplifier a compact polyline and reduces spike opportunities; missing it amplifies boundary error.

### 2.5 Colour fill and corner preservation

Each vtracer cluster is filled with the **arithmetic mean of its residual pixels** (`residue_color() = residue_sum.average()`), a straight per-channel mean with no palette snapping; on a flat region this recovers the exact colour and deltaE is ~0 by construction ([visioncortex cluster.rs](https://github.com/visioncortex/visioncortex/blob/master/src/color_clusters/cluster.rs), confirmed verbatim). If our engine palette-snaps or quantises fills, that is free deltaE we are conceding on every region.

**Corner preservation is a hard angle gate applied before smoothing:** a polygon vertex whose turn angle >= `corner_threshold` (default 60 deg) is flagged a corner and never smoothed away; the 4-point subdivide-smooth switches to a 3-point scheme at corners ([visioncortex smooth.rs](https://github.com/visioncortex/visioncortex/blob/master/src/path/smooth.rs)). Splicing cuts the outline at inflections (turn-sign flips) and wherever accumulated turning reaches `splice_threshold` (45 deg), fitting one flo_curves least-squares Bezier per arc with a `retract_handles` anti-overshoot correction. Uniform smoothing without the corner gate rounds every corner — exactly where rendered deltaE and our label errors concentrate.

Holes are handled explicitly (cluster the negative image, fill interior holes into the parent, emit each hole as its own boundary), so every emitted shape is a simple closed outline rather than relying on even-odd fill ([visioncortex clusters.rs](https://github.com/visioncortex/visioncortex/blob/master/src/clusters.rs), confirmed). *Provenance flag:* the oft-quoted "avoids producing shapes with holes" phrasing was **not** locatable in the repo's root README and may come from external docs — the code behaviour is confirmed regardless.

### 2.6 Clustering mechanics (confirmed, context only)

vtracer's colour mode is **connected-component region-merging, not k-means or palette quantisation**: flood-build same-colour clusters (4/8-connectivity), then merge the smallest-area cluster into its most-similar neighbour until a "deepen" rule promotes it to its own layer ([visioncortex builder.rs](https://github.com/visioncortex/visioncortex/blob/master/src/color_clusters/builder.rs), confirmed). The same-colour test quantises each channel by right-shift `(8 - color_precision)` bits with per-channel tolerance ±1 (default: shift 2, i.e. 6-bit colour, tol 1) ([runner.rs](https://github.com/visioncortex/visioncortex/blob/master/src/color_clusters/runner.rs), confirmed).

> **CORRECTED CLAIM (was REFUTED):** Layer promotion is **not** gated *purely* by colour distance. The original claim ("a cluster becomes its own layer only if its nearest-neighbour L1 RGB diff exceeds layer_difference=16") had the right numbers but the wrong scope. Promotion is a *conjunction*: `patch_good(...) && neighbours[0].diff > deepen_diff`. `patch_good` additionally requires the cluster area to fall in [16, 65536] px and be compact (not thread-like), and a separate `hollow`/neighbour-count condition also applies. **Consequence:** small (<~16px) or thin/thread-like regions will **not** deepen into their own layer even with perfectly separated colours — directly relevant to our known fineline-stroke gap. So the corollary "every distinct region deepens into its own clean layer" is false; the colour test is one of two-plus gates.

### 2.7 Reference defaults (the config we compete against)

`filter_speckle=4` (area 16px dropped), `color_precision=6`, `layer_difference=16`, `corner_threshold=60deg`, `length_threshold=4.0px`, `max_iterations=10`, `splice_threshold=45deg`, `path_precision=2`, `mode=Spline`, `hierarchical=Stacked`, `outset_ratio=8.0` ([vtracer config.rs](https://github.com/visioncortex/vtracer/blob/master/cmdapp/src/config.rs), confirmed). Potrace equivalents where despeckle is `turdsize=2`, `alphamax=1`, `opttolerance=0.2`, `turnpolicy=minority` (the last chosen to keep thin/fine features connected — relevant to our fineline gap). vtracer's `--mode polygon` (straight segments only, no Bezier) is the fair apples-to-apples comparison point for a planar-map engine that emits straight-segment borders before curve fitting.

## 4. Open questions the round could not settle

1. **What does our engine's simplifier and vertex handling actually do today?** The whole ranking assumes we are DP-based / greedy / grid-snapped, inferred from the size of the gap, not from reading our code. Recommendations 1–3 need a ~1-day audit of our own tracer to confirm which levers are genuinely missing versus already present-but-weak. This is the single highest-value next step.

2. **Does our planar-map engine already share edges, or reconcile post-hoc?** Determines whether Rec 3 is a large structural win or a no-op. Not answerable from external sources.

3. **How much of the 0.16 gap is polygon-stage vs curve/Bezier-stage?** The `--mode polygon` ablation (Rec 1's experiment) would split this, but it hasn't been run. If the gap survives at polygon mode, it is simplification; if it only appears at spline mode, it is our Bezier fitting/corner model.

4. **Exact stacked-mode draw order.** Confirmed as hierarchical-build order with genuine overlap, but "largest-area-first" could not be verified against an explicit sort in source. Immaterial to our engine (we can't use stacking on a true partition), but flagged so no ticket over-claims the mechanism.

5. **Whether the 24-image bench can even register the wins.** Mean-deltaE is area-dominated; if several recommendations each move the bench <0.02, we won't be able to rank them. May need a higher-edge-density held-out set (with typography) before fine-grained tuning is worthwhile — which reopens the corpus/typography gap noted in project context.
